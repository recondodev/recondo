# Recondo Documentation Publishing Pipeline

## Overview

The recondo.dev website is a separate Astro-based site hosted at **`/Users/andmer/Projects/recondo-website`**, published to Cloudflare Pages via Wrangler.

This directory (`docs/site/`) in the main recondo repo contains **markdown source** that must be manually synchronized into the website repo for publication. The pipeline is **not automated** — it is an out-of-scope infrastructure task per the Task 1 specification.

## Source Structure

**Main Recondo Repo (source of truth for documentation):**
- Repository: `/Users/andmer/Projects/recondo`
- Docs location: `docs/site/` (this directory)
- Subdirectories:
  - `tui/` — TUI install, first-run, keybindings

- Repository: `github.com/recondodev/website` at `/Users/andmer/Projects/recondo-website`

- Hosting: Cloudflare Pages (via Wrangler)
- Canonical domain: `https://recondo.dev`
- Source location: `src/pages/`

## Build & Deploy Process

### Website Repository

```bash
# Development
npm run dev              # Astro dev server on http://localhost:3000

# Production build
npm run build            # Astro → dist/; adds .assetsignore for Wrangler

# Deployment
npm run deploy           # npm run build && wrangler deploy

# Dry-run (CI step)
npm run check            # npm run build && wrangler deploy --dry-run
```

**Wrangler Configuration** (`wrangler.json`):
- Main worker: `dist/_worker.js/index.js`
- Routes: `recondo.dev` and `www.recondo.dev` (custom domains)
- Asset serving: `dist/` directory
- Observability: enabled (source maps uploaded)

### Recondo Repo → Website Repo Workflow

**Manual sync process (not yet automated):**

1. Write/update markdown files in `/Users/andmer/Projects/recondo/docs/site/`.
2. Move or copy finalized markdown into the website repo's `src/pages/` directory structure (must match the desired URL path).
3. Test on the website repo's dev server (`npm run dev`).
4. Commit to the website repo.
5. Push; Wrangler will automatically deploy on merge to `main`.

**Future automation (out of scope for Task 1):**
- A GitHub Actions workflow in the recondo repo that detects changes to `docs/site/**/*.md`, pushes them to the website repo, and triggers a website deploy.
- Or: a submodule link from the website repo pointing at the recondo repo's `docs/site/`.

## Canonical URL Prefix

**All markdown files should be published under `/docs/` prefix:**

- `docs/site/index.md` → `https://recondo.dev/docs/`
- `docs/site/quickstart.md` → `https://recondo.dev/docs/quickstart/`
- `docs/site/tui/install.md` → `https://recondo.dev/docs/tui/install/`
- `docs/site/mcp/tool-catalog.md` → `https://recondo.dev/docs/mcp/tool-catalog/`

(Astro's `src/pages/` auto-converts markdown filenames to route slugs; directory structure becomes part of the path.)

## Tool Catalog Generation

**Decision: Checked in, regenerated by CI.**

### Behavior

- **Source:** `mcp/scripts/generate-tool-catalog.ts` (to be implemented in Task 6)
- **Output:** `docs/site/mcp/tool-catalog.md`
- **Method:** Script walks the MCP tool registry, emits markdown with one section per tool group
- **Checked in:** Yes — `tool-catalog.md` is committed to the recondo repo
- **CI validation:** `just docs-tool-catalog-check` regenerates into a temp file and diffs against the checked-in version; fails CI if they diverge
- **Developer workflow:** 
  - Run `just docs-tool-catalog` locally to regenerate
  - If the file changes, commit the updated version
  - If CI fails on a PR, regenerate and commit

### Justfile Recipes

```bash
just docs-tool-catalog        # Generate: pnpm --filter recondo-mcp tsx scripts/generate-tool-catalog.ts > docs/site/mcp/tool-catalog.md

just docs-tool-catalog-check  # CI check: pnpm --filter recondo-mcp tsx scripts/generate-tool-catalog.ts | diff -u docs/site/mcp/tool-catalog.md -
```

### Why Checked In?

1. **Human-readable diffs** — tool addition/removal is visible in pull requests
2. **CI gate ensures freshness** — developers cannot merge stale catalogs
3. **Serves as documentation of generator contract** — the expected output lives in git, making the script's behavior testable
4. **Fast website builds** — no generation step at publish time

## Integration Notes

### For Website Maintainers

- When markdown files are added to `docs/site/` in the recondo repo, manually copy them into `website/src/pages/docs/` and test locally.
- Rebuild the website and deploy via `npm run deploy` after each documentation update.
- The sitemap is auto-generated by `@astrojs/sitemap` integration; no manual sitemap updates needed.

### For Recondo Developers

- Write all documentation in `docs/site/` in the recondo repo.
- Use relative markdown links (e.g., `[quickstart](./quickstart.md)`) — Astro will resolve them correctly.
- External links to code (e.g., GitHub repo URLs) should use absolute URLs.
- Test links and rendering in the website repo's dev server before marking ready.

### Deployment Guardrails

- Website deploys happen via `wrangler deploy` (Cloudflare Pages integration).
- No manual DNS or custom domain configuration needed — both `recondo.dev` and `www.recondo.dev` are routed by Wrangler.
- Build logs and deploy history visible in Cloudflare dashboard.
